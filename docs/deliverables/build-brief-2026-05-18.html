Introduction

This brief explains exactly what to build on QuoteMate this week, why each piece matters, what already exists in the system, and the steps involved. It is written in plain English so both the engineering side and John can follow it. It uses the same Work Package numbering as the Revised Word/PDF version, so WP1–WP9 mean exactly the same thing in every file.

The two goals

Everything on the list serves one of two goals that John says will sell this product:

  1. Quotes that are accurate every single time. A tradie will not trust the tool if a price is wrong even once.
  2. The "wow" moment. The customer sees the actual product placed onto a photo of their own kitchen or wall.

The golden rule: order matters

Some items sit on top of others. If they are built in list order, work gets wasted. Build in this order:

Correct build order

1. Fix the accuracy foundation (WP1).  2. Build the operator-owned product catalogue (WP2).  3. Lock jobs to fixed bills of materials and expand the job list (WP3).  4. Use the catalogue for exact-product images (WP4) and supply-mode pricing (WP5).  5. Add the customer-control flows: calendar/urgency (WP6), follow-up (WP7), mid-chat options (WP9).  6. Cost reporting (WP8) runs in parallel.

The one trap to remember all week

Read this before touching pricing

There is a safety check in the code (call it the "price-checker"). It rejects any price it cannot trace back to the database, and sends that quote to the paid inspection instead.

So whenever you add a new place that prices come from (especially WP2), you must teach the price-checker about it in the same change. If you forget, every new quote gets thrown to inspection — the opposite of what John wants.

WP1 — Estimation consistency and accuracy (item 1)

Introduction

This is the first item John listed and the foundation for everything else. If the same tradie request can produce different answers, the product loses trust. The call also raised a real bug (Jeph: "some pricing didn't have a tenant id… messed up after we set up a new tenant") — it lives here because it directly breaks accuracy.

What it is

Make the estimator give the same grounded answer for the same input, and remove every silent way it can use the wrong price.

What's there now

QuoteMate already has shared jobs, shared materials, similar-past-quote retrieval, and a price-checker. But a price book can have a blank or wrong tradie attached, and when none is found the system silently uses another tradie's book or a hard-coded default markup — with no error shown. A plumber could be quoted on electrician numbers and nobody would know.

What you build

  1. An audit script: list every price-book row with no tradie, every active tradie with no book, and any cross-tradie fallback path.
  2. A migration that attaches orphan rows to the correct tradie (or deletes dead ones), then makes the tradie column required so it cannot recur.
  3. A hard rule: if the price book does not match this tradie, stop and route to inspection with a logged reason — never silently fall back. Remove the "grab the oldest book" fallback.
  4. A small accuracy/consistency test set for this week's target jobs, so later changes can be measured against the same baseline.

Watch out

Do this before calling anything else "accurate". New catalogues and renders are only useful once the price foundation is trustworthy.

Done when

The same clean request repeatedly returns the same grounded price, and a misconfigured tradie produces a clear error instead of a silent wrong price.

WP2 — Materials pricing book, brand priming & range pricing (items 2, 3, 4) — THE KEYSTONE

Introduction

These are separate bullets on John's list, but they belong in one build because they all describe the same operator-owned product catalogue.

What it is

Give each operator their own product list, including the brand they prefer and the exact series/range they sell (Clipsal 2000 vs Iconic), with photos and their own prices.

What's there now

The codebase has shared materials and a tenant brand preference, but no full operator-owned catalogue: no range/series, no supplier, no product photo, and no per-operator pricing.

What you build

  1. Create a tenant-owned materials/products table rather than relying only on shared rows.
  2. Store category, brand, range/series, supplier, active flag, product photo, and operator-specific prices.
  3. Extend preference handling so an operator can prefer both brand and range (e.g. Clipsal Iconic → Better, 2000 → Good).
  4. Update material lookup so tenant-owned products reach the estimator and are preferred ahead of generic shared rows.
  5. Extend the price-checker so it accepts prices from this new table. (This is the trap from the Introduction — not optional.)
  6. Keep the data model compatible with WP4 (rendering) and WP9 (mid-process options).

Watch out

This is the keystone. If it is poorly modelled now, WP4, WP5 and WP9 all become messy. And if the price-checker is not extended in the same change, every branded quote gets dumped to inspection.

Done when

An operator can configure the products they actually use, the estimator picks the correct brand and range and it passes the price-checker, and the same catalogue can drive both the render and the mid-chat options.

WP3 — Fixed bills of materials + accurate job-type sourcing (items 5, 6)

Introduction

John wants the same job to price the same way every time, and a broader list of standard jobs so more requests can be handled confidently instead of pushed to inspection.

What it is

Define each standard job as a structured record plus a fixed list of required materials and quantities.

What's there now

A job/assembly catalogue exists (shared + tenant custom), but the bill of materials is loose prompt guidance, not a structured list — so the AI re-decides the parts every time, which is why prices wobble.

What you build

  1. A structured job-to-materials table with quantities and required vs optional parts.
  2. Build quote lines from that table instead of letting the model freely decide each time.
  3. Import John's researched job list (Perplexity / Gemini / Firecrawl) only after validating the source data and mapping it to the correct trade.
  4. Repeat-run tests on representative jobs so bill-of-materials and total price stay stable over time.

Watch out

Sourcing the job list and building the software are different jobs — bad source data only makes wrong quotes happen more consistently. Also tell John: adding jobs does not "break the AI's learning" — there is no learning, the AI just reads the database.

Done when

A standard job produces the same required materials and a stable price across repeated runs, and new job types import without destabilising existing ones.

WP4 — AI render of the exact product on the customer's photo (item 7) — the "wow"

Introduction

This is the second thing John said will sell the product: the customer sees the actual quoted product on their own sink, wall, ceiling or room.

What it is

Use the customer's uploaded photo as the scene and the selected catalogue product as the visual reference.

What's there now

The app already generates Gemini previews from customer photos, but the product is text-driven (for example "Caroma Liano toilet") so Gemini only guesses what it looks like.

What you build

  1. Link quote line items back to the selected catalogue product.
  2. Pass the product photo together with the customer's scene photo to the image step, with the instruction "match this exact product".
  3. Use the selected product consistently across the preview and the sample images.
  4. Add checks so a quoted Clipsal Iconic product does not render as a generic or mismatched fitting.

Watch out

This depends on WP2. Without a real product catalogue photo, the model can only guess. Google Slides was a side comment on the call; ignore it, it is not needed.

Done when

A customer looking at a tap, toilet, downlight or power point preview can recognise the same product that is being quoted.

WP5 — Different pricing when the customer supplies the product (item 8)

Introduction

John specifically called out that some jobs should price differently when the customer supplies the item versus when the tradie supplies it.

What it is

Support two supply modes for the same job: tradie-supplied and customer-supplied.

What's there now

Shared materials carry some metadata, but there is no clear operator-level price difference between supply-and-install and install-only.

What you build

  1. Add separate price fields for tradie-supplied and customer-supplied scenarios in the operator catalogue (WP2).
  2. Capture the supply mode during intake or selection and carry it into the quote calculation.
  3. Change quote wording so the customer can see when they are supplying the item ("Customer to supply [item] — must meet safety standards").
  4. Make sure rendering (WP4) still uses the selected product when the customer has chosen one.

Watch out

Customer-supplied does not mean price-free. Labour, risk and any safety wording still need to be explicit.

Done when

The same product can produce the correct supply-and-install price or install-only price, and the quote clearly explains which scenario was used.

WP6 — Calendar, urgency & lock-in after deposit (item 9)

Introduction

John wants the customer to feel urgency and to lock in either the inspection or the job after paying — so they commit instead of drifting away.

What it is

A flow that moves a customer from quote to held offer to booked work, with visible timing and a confirmed commitment.

What's there now

The app already has payment, booking pages, scheduled-at data and confirmation messages. It does not yet expose an urgency / held-price model, and paying the deposit does not reserve and lock a slot.

What you build

  1. Define the urgency rule — for example, how long the quote or inspection price is held.
  2. Show the hold / expiry clearly on the quote and in messages. This "price held until [date]" countdown is the cheap win — it needs no calendar — so do it first.
  3. Model the handoff after deposit so payment leads into a clear booked or reserved state.
  4. Extend availability handling only as far as the selected week's booking flow needs, rather than building a full calendar product all at once.

Watch out

A countdown without a real business rule is only decoration — decide what is actually being held and what happens when it expires. The full calendar is a multi-week job.

Done when

A customer can see the urgency, pay, and move into a clear reserved / booked state without the tradie manually patching the process together.

WP7 — Setter function for non-conversions (item 10)

Introduction

John wants a VA to follow up with people who received a quote but did not accept it.

What it is

A reliable follow-up queue for quotes that were sent but not converted.

What's there now

Some quote status fields exist, but there is no operational queue, and status is not reliably recorded (the data shows 0 accepted quotes) so today you cannot tell who did not accept.

What you build

  1. Make the quote lifecycle reliable enough to distinguish sent, viewed, accepted and paid.
  2. Add a needs-follow-up view filtered by age and non-conversion status.
  3. Show the customer's details, quote summary, last activity and a direct path to contact them.
  4. Leave room for later automation, but make the human VA workflow useful first.

Watch out

The follow-up list is only as good as the underlying status events. Fix event reliability before trusting the queue.

Done when

A VA can open the dashboard and immediately see which customers received quotes, did not accept, and should be contacted.

WP8 — Cost per transaction / quote (item 11)

Introduction

John wants to understand what each quote costs to produce so the business can see whether the workflow is profitable.

What it is

Roll up the variable cost of generating each quote into one stored figure with a component breakdown.

What's there now

The system performs AI, image, SMS and sometimes voice work, but there is no per-quote profitability number; only AI usage is logged, never converted to dollars.

What you build

  1. Record the usage units needed for AI, image generation, SMS and voice.
  2. Apply the current provider rates to convert those units into dollars. Note each quote makes about four image calls (1 preview + 3 samples) — a real cost, not a rounding error.
  3. Store the total and the component breakdown against the quote.
  4. Expose the cost in the dashboard so John can compare quote value against quote cost.

Watch out

Keep provider rates centralised so future price changes do not require hunting through several code paths. Low risk and additive — do it in parallel.

Done when

Each quote has a visible total production cost and a useful breakdown by cost driver.

WP9 — SMS / MMS to the client mid-process (item 12)

Introduction

John asked for the customer to receive product photos or options during the conversation, not only after the quote is built — "do you want the Clipsal 2000 or the Clipsal Iconic?"

What it is

Show real operator-approved product options mid-conversation and record the customer's choice.

What's there now

The app already supports SMS flows, inbound MMS, photo-request links and WhatsApp fallback. What is missing is an outbound product-option step; today it sends text only.

What you build

  1. Define the moments in the conversation where a product choice should be offered.
  2. Use the operator-owned catalogue from WP2 as the only source for the images and labels shown.
  3. Choose the outbound channel — image message, or SMS with a choice page — based on what is reliable for the tenant's number. Real MMS does not work on Australian Twilio numbers.
  4. Save the customer's selected option so it drives both quote pricing (WP2) and rendering (WP4).

Watch out

Do not show generic products the tradie does not actually sell. Do not treat this as finished until the selected option flows through to both price and preview. Needs WP2 first.

Done when

The customer can choose between real products in the conversation, and the chosen product is the same one used in the quote and the preview.

Additional call items (not on John's numbered list)

  • Rough-range estimate when an exact price is not possible — "this usually costs $200–$400, pay $99 to confirm" — instead of a cold "book an inspection". The similar-past-quote machinery already exists to reuse.
  • $199 → $99 inspection price. Hard-coded in several places; find every occurrence, then move it to one setting.
  • Simplify tiers Good / Better / Best → Good / Better. John said three options are confusing and not always right. Keep the old data column (do not delete history).
